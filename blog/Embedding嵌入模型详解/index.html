Embedding嵌入模型详解

2025年12月29日 · 阅读需 17 分钟
GuguLH
Back-end engineers

前言​

在自然语言处理和大模型应用中，Embedding（嵌入）是一个基础且核心的概念。无论是语义搜索、RAG应用还是推荐系统，都离不开Embedding技术。本文将介绍Embedding的概念、价值、运算法则，以及LangChain中Embedding组件的使用方法。

1. 什么是Embedding​

1.1 Embedding的定义​

Embedding，中文常译为"嵌入"或"向量化"，是一种将离散数据（如文字、图片、用户ID等）转换为连续向量表示的技术。

简单来说，Embedding就是把人类能理解的"内容"转换成计算机能理解的"数字列表"。

举个例子，当我们用Embedding模型处理"我喜欢快乐编程"这句话时，它会生成一个类似这样的向量：

[0.123, -0.456, 0.789, 0.234, -0.567, 0.891, ...]

这个向量可能包含几百甚至几千个维度，每个维度都是一个浮点数。虽然这些数字对人类来说没有直观意义，但它们以数学形式"编码"了这句话的语义信息。

1.2 Embedding的本质​

Embedding的核心思想是：语义相近的内容，其向量表示在空间中也相近。

想象一个三维空间：

  • "猫"和"小猫"的向量位置很接近
  • "猫"和"狗"的向量位置也比较接近（都是动物）
  • "猫"和"汽车"的向量位置则相距较远

这种空间表示使得计算机能够"理解"词语之间的语义关系，进而实现语义搜索、相似度计算等功能。

1.3 Embedding与传统编码的区别​

特性传统编码（如One-Hot）Embedding嵌入
维度等于词表大小，可能非常大固定维度，通常几百到几千
稀疏性高度稀疏（大部分是0）稠密向量（每个维度都有值）
语义信息无语义，只是标识符包含丰富的语义信息
相似度无法计算相似度可以计算向量相似度
泛化能力无法处理未见过词汇可以泛化到相似内容

2. Embedding带来的价值​

2.1 语义理解能力​

Embedding最大的价值在于让计算机具备了"语义理解"能力。

传统关键词匹配的局限：

假设我们要搜索关于"猫咪"的内容：

  • 用户搜索："猫咪"
  • 文档内容："我家的小猫很可爱"

传统关键词匹配会失败，因为"猫咪"和"小猫"是两个不同的词。但Embedding可以理解它们的语义相近，从而成功匹配。

语义相似度计算：

有了Embedding，我们可以量化两个文本的语义相似程度：

文本A文本B相似度
我叫GuguLH，我喜欢快乐编程这个喜欢快乐编程的人叫GuguLH0.95（很高）
我叫GuguLH，我喜欢快乐编程求知若渴，虚心若愚0.45（较低）

2.2 支持多种应用场景​

语义搜索

不再依赖关键词匹配，而是理解用户的真实意图，返回语义相关的结果。

RAG（检索增强生成）

Embedding是RAG的核心：

  1. 将文档分块并向量化
  2. 将用户问题向量化
  3. 检索与问题向量最相似的文档块
  4. 提供给大模型生成回答

推荐系统

将用户偏好和物品都向量化，通过相似度匹配推荐最相关的内容。

聚类分析

将大量文本向量化后，可以自动发现内容的分组和主题。

去重检测

快速识别语义重复的内容，即使表达方式不同。

2.3 降低计算成本​

相比于处理原始文本，向量运算更加高效：

  • 向量相似度计算复杂度固定，不受文本长度影响
  • 可以利用向量索引加速大规模检索
  • 预先计算好的向量可以复用

3. Embedding的运算​

有了向量表示后，我们需要方法来衡量向量之间的关系。最常用的是余弦相似度。

3.1 余弦相似度​

余弦相似度衡量两个向量方向的相似程度，计算的是向量夹角的余弦值。

3.1.1 计算公式​

cos⁡(A,B)=A⋅B∥A∥⋅∥B∥\cos(A, B) = \frac{A \cdot B}{\|A\| \cdot \|B\|}cos(A,B)=∥A∥⋅∥B∥A⋅B​

其中：

  • A · B 是向量的点积（对应位置相乘再相加）
  • ||A|| 是向量A的模长（各分量平方和的平方根）

3.1.2 计算步骤分解​

第一步：计算点积

点积是两个向量对应位置相乘后求和：

A = [a1, a2, a3, ...]
B = [b1, b2, b3, ...]

A · B = a1×b1 + a2×b2 + a3×b3 + ...

第二步：计算向量模长

模长是向量各分量平方和的平方根：

||A|| = √(a1² + a2² + a3² + ...)
||B|| = √(b1² + b2² + b3² + ...)

第三步：计算余弦相似度

将点积除以两个模长的乘积：

cos(A, B) = (A · B) / (||A|| × ||B||)

3.1.3 取值范围与含义​

余弦相似度的取值范围是 -1 到 1：

值含义
1两个向量方向完全相同（语义完全一致）
0两个向量正交（语义无关联）
-1两个向量方向相反（语义相反）

在实际应用中，大多数Embedding产生的向量都是正值或接近正值，所以相似度通常在 0 到 1 之间。

3.1.4 为什么选择余弦相似度​

不受向量长度影响

余弦相似度只关注向量方向，不关注向量长度。这意味着：

  • 一段长文本和一段短文本，如果表达相同的意思，它们的相似度依然很高
  • 这对于文本处理非常重要，因为文本长度变化很大

计算效率高

点积和模长的计算都是简单的乘加运算，非常适合计算机处理。

适合高维空间

在数百甚至数千维的空间中，余弦相似度仍然有效。

3.2 欧几里得距离​

另一种常用的度量方法是欧几里得距离，它衡量两个向量在空间中的直线距离。

d(A,B)=∥A−B∥2=∑i=1n(ai−bi)2d(A, B) = \|A - B\|_2 = \sqrt{\sum_{i=1}^{n} (a_i - b_i)^2}d(A,B)=∥A−B∥2​=i=1∑n​(ai​−bi​)2​

与余弦相似度不同：

  • 欧几里得距离越小，表示越相似
  • 欧几里得距离受向量长度影响

3.3 两种方法的对比​

特性余弦相似度欧几里得距离
值范围-1 到 10 到 ∞
相似判断值越大越相似值越小越相似
长度敏感不敏感敏感
适用场景文本语义相似度图像、空间位置
常用程度文本处理首选特定场景使用

4. LangChain中的Embedding组件及其使用​

LangChain提供了统一的Embedding接口，支持多种嵌入模型，包括OpenAI、HuggingFace等。下面介绍几种常用的Embedding组件。

4.1 OpenAI Embedding​

4.1.1 基本使用​

创建嵌入模型实例

使用OpenAI的嵌入模型需要提供API密钥和基础URL：

from langchain_openai.embeddings import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(
    base_url=BASE_URL,
    api_key=API_KEY,
    model=DEFAULT_MODEL,
)

嵌入单个文本

使用embed_query方法将单个文本转换为向量：

vector = embeddings.embed_query("我叫GuguLH，我喜欢快乐编程")

返回的是一个浮点数列表，长度取决于模型（如1536维或3072维）。

嵌入多个文档

使用embed_documents方法批量处理多个文本：

vectors = embeddings.embed_documents([
    "我叫GuguLH，我喜欢快乐编程",
    "这个喜欢快乐编程的人叫GuguLH",
    "求知若渴，虚心若愚",
])

返回的是一个向量列表，每个元素对应一个输入文本的向量。

4.1.2 参数说明​

参数说明
base_urlAPI服务地址，可用于对接兼容接口
api_keyAPI密钥
model使用的模型名称，如text-embedding-3-small
check_embedding_ctx_length是否检查文本长度限制

4.2 CacheBackEmbedding：带缓存的嵌入​

需要注意的是，CacheBackEmbedding不是一个独立的嵌入模型，而是一个包装器（Wrapper）。它的作用是为任意嵌入模型添加缓存能力：

原始模型 → CacheBackEmbedding包装 → 带缓存能力的模型

组合示例：

  • OpenAI + CacheBackEmbedding = 带缓存的OpenAI嵌入
  • HuggingFace + CacheBackEmbedding = 带缓存的本地嵌入

在实际应用中，相同的文本可能需要多次计算向量，这会带来：

  • 重复的API调用费用
  • 重复的等待时间
  • 不必要的资源消耗

LangChain提供了CacheBackedEmbeddings组件来解决这个问题。

4.2.1 缓存的工作原理​

第一次请求："我喜欢编程" → 调用API → 返回向量 → 存入缓存
第二次请求："我喜欢编程" → 检查缓存 → 命中缓存 → 直接返回向量

4.2.2 基本使用​

创建带缓存的嵌入模型

# 创建底层嵌入模型
embeddings = OpenAIEmbeddings(
    base_url=BASE_URL,
    api_key=API_KEY,
    model=DEFAULT_MODEL,
)

# 包装成带缓存的版本
embeddings_with_cache = CacheBackedEmbeddings.from_bytes_store(
    embeddings,
    LocalFileStore("./cache"),      # 本地文件存储
    query_embedding_cache=True,      # 缓存查询向量
    key_encoder=sha256_encoder,      # 键编码器
)

存储位置

缓存可以存储在不同位置：

  • LocalFileStore：本地文件系统
  • InMemoryStore：内存（重启后丢失）
  • 自定义存储：实现标准接口即可

4.2.3 键编码器​

键编码器决定如何将文本转换为缓存的键：

def sha256_encoder(payload: str) -> str:
    return hashlib.sha256(payload.encode()).hexdigest()

使用SHA256哈希可以：

  • 生成固定长度的键
  • 避免文件名中的特殊字符问题
  • 保证相同文本生成相同的键

4.2.4 缓存的优势​

优势说明
节省费用避免重复调用付费API
提升速度本地读取比网络请求快得多
可复现相同输入保证相同输出
离线可用预先缓存后可离线使用

4.2.5 注意事项​

缓存失效问题

当底层模型更新后，旧缓存可能与新模型不兼容。解决方法：

  • 使用namespace参数区分不同模型
  • 更新模型时清除旧缓存

存储空间

大量文本的缓存会占用磁盘空间，需要定期清理。

一致性

确保缓存键的生成方式一致，否则可能导致缓存未命中。

4.3 HuggingFace嵌入模型​

HuggingFace提供了丰富的开源嵌入模型，可以本地部署或使用远程推理服务。

4.3.1 本地部署​

本地部署适合：

  • 对数据隐私有要求的场景
  • 需要离线运行的应用
  • 想要节省API费用的项目

基本使用

embeddings = HuggingFaceEmbeddings(
    cache_folder="./embeddings/"  # 模型缓存目录
)

默认会下载sentence-transformers/all-mpnet-base-v2模型，这是一个效果不错的开源嵌入模型。

工作流程：

  1. 首次运行时从HuggingFace下载模型
  2. 模型保存在本地缓存目录
  3. 后续使用直接加载本地模型

优势：

  • 无API调用费用
  • 数据不出本地
  • 可自定义模型

局限：

  • 需要本地计算资源
  • 首次下载模型需要时间
  • 效果可能不如顶级商业模型

4.3.2 远程推理服务​

HuggingFace也提供远程推理API，无需本地部署：

embeddings = HuggingFaceEndpointEmbeddings(
    huggingfacehub_api_token=API_TOKEN,
)

优势：

  • 无需本地计算资源
  • 快速开始使用
  • 可使用更强大的模型

局限：

  • 需要API Token
  • 有调用费用（部分模型免费）
  • 依赖网络连接

4.4 Embedding模型对比​

上文介绍了三种嵌入模型，它们各有特点：

模型部署方式费用效果适用场景
OpenAI Embedding云端API按量付费优秀生产环境、追求效果
HuggingFace本地本地部署免费良好隐私要求、离线场景
HuggingFace远程云端API部分免费良好快速原型、中小项目

4.5 统一的接口设计​

LangChain的所有Embedding组件都实现了统一的接口：

方法说明
embed_query(text)将单个文本转换为向量
embed_documents(texts)将多个文本批量转换为向量

这种统一设计使得：

  • 不同组件可以轻松切换
  • 代码更加模块化
  • 便于测试和维护

5. 总结​

5.1 核心要点回顾​

  1. Embedding是将内容转换为向量表示的技术，让计算机能够"理解"语义

  2. Embedding的价值在于支持语义搜索、RAG、推荐系统等多种AI应用

  3. 余弦相似度是最常用的相似度计算方法，关注向量方向而非长度

  4. LangChain提供多种Embedding组件：

    • OpenAI Embedding：效果优秀，适合生产环境
    • HuggingFace：开源免费，可本地部署
    • CacheBackEmbedding：包装器组件，为任意模型添加缓存能力

5.2 选型建议​

需求推荐方案
追求最佳效果OpenAI Embedding
数据隐私要求HuggingFace本地部署
控制成本HuggingFace本地部署
重复查询多任意模型 + CacheBackEmbedding
快速原型HuggingFace远程API

5.3 最佳实践​

  1. 选择合适的维度：维度越高信息越丰富，但计算成本也越高
  2. 使用缓存：对于重复文本，缓存能显著节省费用和时间
  3. 关注模型版本：不同版本的模型可能产生不同的向量
  4. 批量处理：使用embed_documents批量处理比循环调用embed_query更高效
