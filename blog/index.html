11. Vue 生命周期函数​

  • beforeCreate: 调用app.mount()创建 Vue 实例之后, createApp()中配置项生效之前调用
  • created: 在beforeCreate()之后,createApp()中配置项生效之前调用,这个时候 computed,method 和 watch 已经配置好了
  • beforeMount: 在 created 生命周期执行完成之后,应用还没有挂载到app.mount()指定的 html 元素上,在挂载之前会调用beforeMount()生命周期钩子,这个时候应用还没在页面上渲染出来
  • mounted: mounted()会在应用挂载到app.mount()指定的 html 元素后执行,这个时候应用已经在页面上渲染出来了
  • beforeUpdate: 当应用中 html 模板需要重新渲染时,例如 data 中的属性发生了变化,那么在刷新之前会执行beforeUpdate()生命周期钩子
  • updated: updated()是在数据更新之后,html 重新渲染完成之后调用
  • beforeUnmount: beforeUnmount()会在应用卸载前执行,调用app.unmount()会卸载组件,那么正式卸载前,先执行beforeUnmount()生命周期钩子,这个时候 Vue 还是正常的,可以在里面做一些清理收尾操作
  • unmounted: unmounted()生命周期会在应用正式卸载之后调用,这个时候跟应用有关的事件监听,或者指令绑定都已经卸载了
  • 常用的生命周期钩子

5. 定义 Controller,完成简单对话​

    @Autowired
    private ChatClient chatClient;

    /**
     * 阻塞式调用
     */
    @GetMapping("/chat")
    public String chat(@RequestParam(name = "prompt", required = true) String prompt) {
        return chatClient.prompt()
                .user(prompt)
                .call() // 阻塞式调用
                .content();
    }

    /**
     * 使用WebFlux完成流式响应,前端类似打字机效果
     */
    @GetMapping(value = "/chat-stream", produces = "text/html;charset=utf-8")
    public Flux<String> chatStream(@RequestParam(name = "prompt", required = true) String prompt) {
        return chatClient.prompt()
                .user(prompt)
                .stream() // 流式响应
                .content();
    }

Gin 框架：Middleware 中间件的艺术与实践

2024年4月7日 · 阅读需 9 分钟
GuguLH
Back-end engineers

前言​

在构建高性能、模块化的 Web 应用时，中间件（Middleware）是不可或缺的基石。对于 Go 语言的明星级 Web 框架 Gin 而言，中间件机制的简洁和强大，是其广受开发者喜爱的重要原因之一。本文将从 Go 语言程序员的角度，深入剖析 Gin 中间件的原理、应用场景，并以一个实用的跨域（CORS）中间件为例，给出完整的代码实践。

1. 什么是 Gin 中间件？​

在 Gin 框架中，中间件本质上是一个或多个处理器函数（HandlerFunc），它们可以在 HTTP 请求到达最终的业务逻辑处理函数（Handler）之前或之后执行预定的逻辑。

Gin 框架的核心设计采用了洋葱模型（Onion Model）或称为链式处理（Chain of Handlers）。当一个请求进入 Gin 路由时，它会依次穿过一系列的中间件，直到到达最终的业务 Handler。

1.1 中间件的结构​

在 Gin 中，一个中间件就是一个标准的 gin.HandlerFunc：

type HandlerFunc func(*Context)

这个函数接收一个指向 gin.Context 的指针，Context 对象包含了请求的所有信息，并提供了修改响应、控制流程的方法。

1.2 核心方法：c.Next()​

Gin 中间件的关键在于 c.Next() 方法。

  • 执行 c.Next() 之前：执行的逻辑被称为 前置处理（Before），例如日志记录、请求参数校验。
  • 调用 c.Next()：将控制权移交给链中的下一个中间件或最终的 Handler。
  • 执行 c.Next() 之后：执行的逻辑被称为 后置处理（After），例如响应修改、耗时统计。

这种设计使得中间件可以“包裹”业务逻辑，形成了强大的功能扩展能力。

2. Gin 中间件的工作流程 (洋葱模型)​

Gin 的洋葱模型是理解中间件执行顺序的关键。请求从最外层（第一个中间件）进入，层层深入，直到核心（最终 Handler）。处理完成后，再从核心一层层返回，执行后置逻辑。

2.1 常见的应用场景​

Gin 中间件可以应用于各种横切关注点：

  1. 身份认证/授权：检查 JWT 或 Session，判断用户是否有权限访问。
  2. 日志记录：记录请求的路径、方法、IP 地址和响应时间。
  3. 跨域处理 (CORS)：设置必要的 HTTP 响应头，允许跨域请求。
  4. 数据校验：在到达业务层前，统一对请求参数进行结构和内容的校验。
  5. 限流/熔断：控制请求频率，保护后端服务。
  6. 错误恢复：使用 gin.Recovery() 捕获 panic，避免服务崩溃。

3. 实践：构建一个通用的 CORS 跨域中间件​

跨源资源共享（CORS）是 Web 开发中几乎必不可少的配置。Gin 框架虽然没有内置，但实现一个中间件非常简单。当然也可以使用第三方库: github.com/gin-contrib/cors

3.1 CORS 中间件代码示例​

这个中间件将处理浏览器的预检请求（Preflight Request，即 OPTIONS 方法），并设置必要的响应头，允许客户端进行跨域请求。

package middleware

import (
    "net/http"
    "github.com/gin-gonic/gin"
)

// CorsMiddleware 是一个通用的跨域处理中间件
func CorsMiddleware() gin.HandlerFunc {
    return func(c *gin.Context) {
        // 1 设置允许所有域访问
        // 在生产环境中，建议替换为特定的域名，例如：c.Writer.Header().Set("Access-Control-Allow-Origin", "[https://yourdomain.com](https://yourdomain.com)")
        c.Writer.Header().Set("Access-Control-Allow-Origin", "*")

        // 2 允许的请求方法
        c.Writer.Header().Set("Access-Control-Allow-Methods", "POST, GET, OPTIONS, PUT, DELETE, UPDATE")

        // 3 允许的请求头
        // 必须包含 Authorization 以支持 JWT 等认证方式
        c.Writer.Header().Set("Access-Control-Allow-Headers", "Origin, Content-Type, Content-Length, Accept-Encoding, X-CSRF-Token, Authorization")

        // 4 允许携带认证信息（如 Cookies、HTTP 认证信息）
        // 注意：如果 Access-Control-Allow-Origin 设置为 "*"，则不能设置 AllowCredentials 为 true
        // c.Writer.Header().Set("Access-Control-Allow-Credentials", "true")

        // 5 处理浏览器发送的预检请求 (OPTIONS)
        // 如果是 OPTIONS 请求，直接返回 204 No Content，并中止后续处理
        if c.Request.Method == "OPTIONS" {
            c.AbortWithStatus(http.StatusNoContent)
            return
        }

        // 6 执行后续的中间件和最终的业务 Handler
        c.Next()
    }
}

3.2 中间件的注册与使用​

在 Gin 应用中，中间件可以在全局、路由分组或单个路由级别进行注册。

package main

import (
    "net/http"
    "github.com/gin-gonic/gin"
    "your_project_name/middleware" 
)

func main() {
    r := gin.New()

    // 1 全局注册中间件
    // 所有请求都会经过这两个中间件
    r.Use(gin.Recovery()) // Gin 内置的错误恢复中间件
    r.Use(middleware.CorsMiddleware()) // 注册我们的 CORS 中间件

    // 2 注册路由
    r.GET("/ping", func(c *gin.Context) {
        c.JSON(http.StatusOK, gin.H{
            "message": "pong",
        })
    })

    // 3 路由分组注册中间件
    // 只有 /api/v1 下的路由会应用 AuthMiddleware
    v1 := r.Group("/api/v1")
    v1.Use(AuthMiddleware())
    {
        v1.GET("/user", func(c *gin.Context) {
            // 业务逻辑...
            c.JSON(http.StatusOK, gin.H{"user": "info"})
        })
    }

    r.Run(":8080")
}

// 模拟一个认证中间件
func AuthMiddleware() gin.HandlerFunc {
    return func(c *gin.Context) {
        token := c.GetHeader("Authorization")
        if token == "" {
            c.AbortWithStatusJSON(http.StatusUnauthorized, gin.H{"error": "Unauthorized"})
            return
        }
        // 验证 token 逻辑...
        c.Next()
    }
}

4. 深入理解：流程控制与中断​

在某些情况下，中间件需要在请求未通过校验时中断后续的执行。Gin 提供了两个关键的方法来实现流程控制：

4.1 c.Abort()​

作用：立即停止执行后续的中间件和 Handler。

注意：c.Abort() 只会阻止 c.Next() 之后的代码执行，但当前中间件 c.Abort() 之后的代码仍然会执行（即后置处理部分）。

func AuthMiddleware() gin.HandlerFunc {
    return func(c *gin.Context) {
        if !isValid(c) {
            c.JSON(http.StatusUnauthorized, gin.H{"error": "Invalid Token"})
            c.Abort() // 中断后续流程
            return
        }
        c.Next() // 如果验证通过，进入下一个环节
    }
}

4.2 c.AbortWithStatusJSON()​

作用：在中止流程的同时，向客户端返回一个 JSON 响应。

注意：这个方法是 c.JSON() 和 c.Abort() 的组合，是处理错误和权限问题的推荐方式。

5. 总结与最佳实践​

Gin 的中间件机制是构建健壮 Web 应用的利器，它实现了职责分离和代码复用。

实践点描述
细化职责一个中间件只做一件事，例如：CORS 只负责跨域头，Auth 只负责认证。
错误处理使用 c.AbortWithStatusJSON() 优雅地处理校验失败和权限拒绝。
性能考量避免在全局注册过多不必要的中间件，尤其是在高性能要求的路由上。
Context 使用通过 c.Set() 和 c.Get() 在中间件之间以及中间件和 Handler 之间传递数据，例如传递解析出的 user_id。

掌握了 Gin 的洋葱模型和 c.Next() 的精髓，你就能轻松地利用中间件来构建出更加优雅、可维护的 Go Web 服务。

JWT 解析：原理、实践与常见问题

2024年4月2日 · 阅读需 9 分钟
GuguLH
Back-end engineers

前言​

JSON Web Token（JWT）已成为现代应用程序中身份认证的通用标准。它以其无状态（Stateless）特性，完美契合了 RESTful API 和微服务架构。本文旨在深入浅出地讲解 JWT 的核心原理、实际应用，并剖析一些开发者在实践中常遇到的问题。

1. 什么是 JWT？​

JWT 的本质是一个遵循特定规范的、紧凑且自包含的 JSON 对象。它被用于在各方之间安全地传输信息。一个完整的 JWT 由三部分组成，用点（.）分隔：

  • Header（头部）：包含令牌的类型（typ）和使用的签名算法（alg）。例如：

    {
      "alg": "HS256",
      "typ": "JWT"
    }

  • Payload（载荷）：包含声明（claims），即一些实体（通常是用户）和附加元数据。声明分为三类：

    • 注册声明（Registered Claims）：预定义的标准字段，如 iss（签发者）、exp（过期时间）、sub（主题）等。
    • 公共声明（Public Claims）：开发者自定义的声明，用于传递公共信息，应避免命名冲突。
    • 私有声明（Private Claims）：用于在特定应用中使用的自定义声明，不建议用于敏感数据。

  • Signature（签名）：由 Base64URL 编码后的 Header、Payload，以及一个密钥（Secret）通过指定的算法（如 HMAC-SHA256）计算得到。它的作用是验证 JWT 是否被篡改。

2. JWT 的工作流程​

JWT 的核心工作流程非常简洁：

  1. 认证：用户使用用户名和密码登录。
  2. 生成令牌：服务器验证凭据后，创建一个包含用户 ID 和其他信息的 JWT，并用密钥对其进行签名。
  3. 返回令牌：服务器将这个 JWT 返回给客户端。
  4. 携带令牌：客户端在后续的每次请求中，将 JWT 放在 HTTP Header 的 Authorization 字段中（通常格式为 Bearer <token>）。
  5. 验证令牌：服务器收到请求后，用相同的密钥验证 JWT 的签名，并解析 Payload 获取用户信息，完成身份认证。

3. 为什么选择 JWT？​

  • 无状态（Stateless）：服务器无需在内存中保存会话信息，减轻了服务器压力，尤其适用于分布式和微服务架构。

  • 跨域兼容：JWT 可以通过 Header 传递，天然支持跨域认证。

  • 自包含：它包含所需的所有信息，服务器无需频繁查询数据库。

4. 实践中的常见问题与最佳实践​

问题一：令牌过期怎么办？​

解决方案：引入 Refresh Token（刷新令牌） 机制。

  • Access Token（访问令牌）：生命周期短（如 15 分钟），用于访问受保护的资源。

  • Refresh Token：生命周期长（如 7 天），用于在 Access Token 过期后，向认证服务器请求新的 Access Token。

这样既保证了安全性，又提升了用户体验。

问题二：如何处理令牌撤销？​

解决方案：

  • 黑名单机制：将已签发但需要提前失效的 JWT 存储在 Redis 或其他缓存中。每次验证时，先检查黑名单。

  • 签发版本号：在 Payload 中加入用户版本号（如 version: 1）。当用户密码修改、权限变更等需要强制所有旧令牌失效时，更新版本号。服务器验证时，如果令牌中的版本号与数据库不一致，则拒绝。

5. 安全性考量与最佳实践​

JWT 虽然强大，但如果使用不当也会引入安全风险。以下几点至关重要：

  1. 保护你的密钥：签名密钥是 JWT 安全的核心。它必须被妥善保管，不能暴露在客户端代码中。建议将密钥存储在环境变量中，或使用专门的密钥管理服务（如 AWS KMS、Vault）。
  2. Payload 并非加密：切记，JWT 的 Payload 只是经过 Base64URL 编码，任何人都可以轻松解码。因此，绝对不要在 Payload 中存放任何敏感信息，如密码、银行卡号等。
  3. 避免使用弱算法：确保你的签名算法是安全的，例如 HMAC-SHA256、RSA-SHA256。要特别注意，避免使用"none"算法，因为这会绕过签名验证，让攻击者可以伪造任意的 Payload。

func copy(dest interface{}, source interface{}) error {
    sType := reflect.TypeOf(source)
    sValue := reflect.ValueOf(source)
    // 如果为指针类型，则获取其值
    if sType.Kind() == reflect.Ptr {
        sType = sType.Elem()
        sValue = sValue.Elem()
    }
    dType := reflect.TypeOf(dest)
    dValue := reflect.ValueOf(dest)
    if dType.Kind() != reflect.Ptr {
        return errors.New("目标对象必须为struct指针类型")
    }
    dType = dType.Elem()
    dValue = dValue.Elem()
    if sValue.Kind() != reflect.Struct {
        return errors.New("源对象必须为struct或struct的指针")
    }
    if dValue.Kind() != reflect.Struct {
        return errors.New("目标对象必须为struct的指针")
    }
    destObj := reflect.New(dType)
    for i := 0; i < dType.NumField(); i++ {
        destField := dType.Field(i)
        if sourceField, ok := sType.FieldByName(destField.Name); ok {
            if destField.Type != sourceField.Type {
                continue
            }
            value := sValue.FieldByName(destField.Name)
            destObj.Elem().FieldByName(destField.Name).Set(value)
        }
    }
    dValue.Set(destObj.Elem())
    return nil
}

func sliceColumn(slice interface{}, column string) interface{} {
    t := reflect.TypeOf(slice)
    v := reflect.ValueOf(slice)
    if t.Kind() == reflect.Ptr {
        t = t.Elem()
        v = v.Elem()
    }
    if v.Kind() == reflect.Struct {
        val := v.FieldByName(column)
        return val.Interface()
    }
    if v.Kind() != reflect.Slice {
        return nil
    }
    t = t.Elem()
    if t.Kind() == reflect.Ptr {
        t = t.Elem()
    }
    f, _ := t.FieldByName(column)
    sliceType := reflect.SliceOf(f.Type)
    s := reflect.MakeSlice(sliceType, 0, 0)
    for i := 0; i < v.Len(); i++ {
        o := v.Index(i)
        if o.Kind() == reflect.Struct {
            val := o.FieldByName(column)
            s = reflect.Append(s, val)
        }
        if o.Kind() == reflect.Ptr {
            v1 := o.Elem()
            val := v1.FieldByName(column)
            s = reflect.Append(s, val)
        }
    }
    return s.Interface()
}

# 将模块下载到本地缓存,需要指定模块路径及版本号
go mod download github.com/gin-gonic/gin@v1.9.0

# 初始化一个新的模块到当前目录(通常使用github等为域名)
go mod init github.com/gugulh/basic-go

# 添加缺少的依赖,删除未使用的依赖
go mod tidy

# 通过工具或脚本编辑go.mod(通常不会使用这个命令进行编辑)
go mod edit -require="github.com/gin-gonic/gin@v1.9.0"

# 根据go.mod中的依赖项制作vendor副本
# 有了vendor副本,项目将不再依赖本地缓存
go mod vendor

# 验证依赖是否正确
go mod verify

# 返回对指定模块的依赖关系最短路径,解释为什么依赖指定包
go mod why github.com/go-playground/validator/v10

3. go install/get/clean​

# go install 安装可执行插件
go install github.com/google/gops@latest

# go get 获取模块信息并更新go.mod文件
# 若本地缓存没有该模块,则下载模块;有则直接引用
go get github.com/gin-gonic/gin@v1.9.0

# go get -u 更新模块依赖,并更新go.mod
go get -u github.com/gin-gonic/gin@v1.9.0

# go clean 清理临时目录中的文件
go clean -modcache

4. go install 和 go get的区别​

二者都用于从远程仓库（如GitHub）下载包。go get侧重于获取和更新包及其依赖，并更新go.mod文件。go install则专注于编译和安装可执行文件，而不会更新go.mod文件。
