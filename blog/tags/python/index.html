LangChain：Model 组件及使用技巧

2025年12月26日 · 阅读需 7 分钟
GuguLH
Back-end engineers

前言​

Model 组件是 LangChain 的核心，它封装了对大语言模型的调用。本文基于 LangChain 1.2，结合官方示例代码，聊聊 Model 组件的几种常见用法。

1. 先跑起来：最基础的用法​

直接看代码：

import os
from datetime import datetime

import dotenv
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI

dotenv.load_dotenv()

# 创建模型实例
llm = ChatOpenAI(
    api_key=os.getenv("API_KEY"),
    base_url=os.getenv("BASE_URL"),
    model=os.getenv("DEFAULT_MODEL"),
)

# 编排 Prompt
prompt = ChatPromptTemplate.from_messages(
    [
        ("system", "你是OpenAI开发的聊天机器人,请回答用户的问题,现在的时间是{now}"),
        ("human", "{query}")
    ]
).partial(now=datetime.now)

# 调用模型
ai_message = llm.invoke(prompt.invoke({
    "query": "现在是几点?我现在应该做什么?",
}))
print(ai_message.content)

输出示例：

现在是下午3点45分。根据您目前的时间，我建议您可以：

1. 喝杯咖啡休息一下
2. 处理一些需要集中精力的工作
3. 站起来活动活动

有什么我可以帮助您的吗？

几点说明：

  • ChatOpenAI 是 OpenAI 模型的封装，来自 langchain_openai 包
  • invoke() 是核心方法，接收 PromptValue 对象，返回 AIMessage
  • ai_message.content 是模型的文本回复
  • ai_message.response_metadata 包含响应元数据，如 token 使用量、模型名称等

1.1 response_metadata 详解​

print(ai_message.response_metadata)

输出示例：

{
    'token_usage': {'completion_tokens': 86, 'prompt_tokens': 58, 'total_tokens': 144},
    'model_name': 'gpt-3.5-turbo',
    'system_fingerprint': 'fp_1234567890',
    'finish_reason': 'stop',
    'logprobs': None
}

这个元数据对于监控模型使用、调试应用非常有用。

2. 批处理：一次处理多个请求​

如果需要同时处理多个请求，用 batch() 方法可以显著提升效率：

# 准备多个请求
requests = [
    prompt.invoke({"query": "你好,你是?"}),
    prompt.invoke({"query": "请讲一个关于程序员的冷笑话"}),
    prompt.invoke({"query": "用一句话介绍Python")],
]

# 批量调用
ai_messages = llm.batch(requests)

for ai_message in ai_messages:
    print(ai_message.content)
    print("---------------------------")

输出：

你好！我是由OpenAI开发的AI助手，可以帮助您回答各种问题。有什么我可以帮助您的吗？
---------------------------
为什么程序员总是分不清万圣节和圣诞节？

因为 Oct 31 = Dec 25

（Octal 31 = Decimal 25）
---------------------------
Python是一门简洁优雅的高级编程语言，以可读性和易用性著称。
---------------------------

优势对比：

方式耗时适用场景
循环调用 invoke()O(n) × 单次耗时请求少、无并发需求
batch() 批量调用O(1) 或 O(批次)多个独立请求、需要批量处理

batch() 的原理是将多个请求打包发送给模型服务商，由服务商内部并行处理，返回结果列表。

3. 流式输出：实时看到生成过程​

对于需要实时反馈的场景，比如聊天机器人，流式输出很有用：

response = llm.stream(prompt.invoke({
    "query": "你能让我极易理解的方式,讲清楚大模型原理吗?"
}))

for chunk in response:
    print(chunk.content, end="", flush=True)

输出是逐字/逐词显示的，用户可以看到模型"思考"的过程，而不是等待完整回答。

流式输出的特点：

  • 返回的是生成器，每次迭代得到一个 AIMessageChunk
  • chunk.content 是本轮的增量内容
  • 适合长文本生成场景，用户可以提前看到部分结果
  • 常用于聊天界面、终端输出等需要实时反馈的地方

3.1 三种调用方式对比​

方法返回类型等待方式适用场景
invoke()AIMessage同步等待完整结果单次请求、简单场景
batch()List[AIMessage]等待所有请求完成批量处理、离线任务
stream()Iterator[AIMessageChunk]流式返回、增量输出聊天机器人、长文本生成

4. 思考内容分离：reasoning_split 参数​

在创建模型时，可以通过 extra_body 设置高级参数：

llm = ChatOpenAI(
    api_key=API_KEY,
    base_url=BASE_URL,
    model=DEFAULT_MODEL,
    extra_body={"reasoning_split": True},  # 将思考内容分离
)

启用后，模型的思考过程会被分离到 reasoning_details 字段，而 content 只包含最终回答：

ai_message = llm.invoke(prompt.invoke({"query": "解释一下为什么天空是蓝色的"}))

print("最终回答:", ai_message.content)
print("思考过程:", ai_message.additional_kwargs.get("reasoning_details"))

这个功能适用于支持思考过程分离的模型（如某些推理模型），方便对思考过程进行审查或展示。

5. 完整示例：结合 Prompt 和 Model​

来看一个完整的例子，串联前面的知识：

import os
from datetime import datetime

import dotenv
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI

dotenv.load_dotenv()

# 1. 编排 Prompt
prompt = ChatPromptTemplate.from_messages(
    [
        ("system", "你是{assistant}，现在的时间是{now}。"),
        ("human", "{query}")
    ]
).partial(
    assistant="小助手",
    now=datetime.now
)

# 2. 创建模型
llm = ChatOpenAI(
    api_key=os.getenv("API_KEY"),
    base_url=os.getenv("BASE_URL"),
    model=os.getenv("DEFAULT_MODEL"),
)

# 3. 方式一：单个请求
ai_msg = llm.invoke(prompt.invoke({
    "query": "推荐一道简单的家常菜"
}))
print(f"单个请求: {ai_msg.content}\n")

# 4. 方式二：批量请求
batch_results = llm.batch([
    prompt.invoke({"query": "什么是Python？"}),
    prompt.invoke({"query": "用三个词形容大海"}),
])
for r in batch_results:
    print(f"批量结果: {r.content}")

# 5. 方式三：流式输出
print("\n流式输出: ", end="")
for chunk in llm.stream(prompt.invoke({"query": "说一句话"})):
    print(chunk.content, end="", flush=True)
print()

6. 总结​

功能方法返回类型场景
单次调用invoke()AIMessage简单请求
批量处理batch()List[AIMessage]并行处理多个请求
流式输出stream()Iterator[AIMessageChunk]实时展示生成过程
思考分离extra_body={"reasoning_split": True}reasoning_details推理模型审查

LangChain 1.2 的 Model 组件设计得很简洁，核心就是 invoke、batch、stream 三个方法。掌握这三个方法，基本就能应对大部分场景了。

LangChain：Prompt 组件及使用技巧

2025年12月26日 · 阅读需 10 分钟
GuguLH
Back-end engineers

前言​

Prompt 是大模型应用的核心入口，LangChain 提供了完善的提示模板管理能力。本文基于 LangChain 1.2，围绕官方示例代码，聊聊怎么用好 Prompt 组件。

1. 先跑起来：最基础的用法​

直接看代码：

from langchain_core.prompts import PromptTemplate

prompt = (
    PromptTemplate.from_template("请讲一个关于{subject}的冷笑话")
    + "，让我开心下"
    + "\n使用{language}语言"
)
print(prompt.invoke({"subject": "程序员", "language": "中文"}).to_string())

输出：

请讲一个关于程序员的冷笑话，让我开心下
使用中文语言

几点说明：

  • PromptTemplate.from_template() 是最常用的创建方式，花括号里的 {subject} 是变量占位符
  • 模板可以用 + 号拼接普通字符串，很灵活
  • invoke() 方法接收字典参数，返回 PromptValue 对象
  • to_string() 把结果转成字符串输出

2. format 和 invoke 的区别​

除了 invoke()，还有个 format() 方法：

from langchain_core.prompts import PromptTemplate

prompt = PromptTemplate.from_template("请讲一个关于{subject}的冷笑话")

# 方式一：format 返回字符串
print(prompt.format(subject="喜剧演员"))
# 输出: 请讲一个关于喜剧演员的冷笑话

# 方式二：invoke 返回 PromptValue
prompt_value = prompt.invoke({"subject": "程序员"})
print(prompt_value.to_string())
print(prompt_value.to_messages())

区别：

方法返回类型适用场景
format()str只需要字符串，不需要后续处理
invoke()PromptValue需要转成消息列表、或者链式调用

PromptValue 是 LangChain 内部统一的消息格式，它可以被转换成字符串或消息列表，适配不同的模型接口。

2.1 to_string() 和 to_messages() 详细对比​

invoke() 返回的 PromptValue 有两个转换方法，用法不同。先看字符串模板：

from langchain_core.prompts import PromptTemplate

str_prompt = PromptTemplate.from_template("请讲一个关于{subject}的冷笑话")
str_value = str_prompt.invoke({"subject": "程序员"})

print("=== PromptTemplate ===")
print("to_string():")
print(str_value.to_string())
print()
print("to_messages():")
print(str_value.to_messages())

输出：

=== PromptTemplate ===
to_string():
请讲一个关于程序员的冷笑话

to_messages():
[HumanMessage(content='请讲一个关于程序员的冷笑话')]

再看聊天模板：

from langchain_core.prompts import ChatPromptTemplate

chat_prompt = ChatPromptTemplate.from_messages([
    ("system", "你是{assistant}，当前时间是 {now}"),
    ("human", "我喜欢 {hobby}，请推荐一部相关电影"),
])

chat_value = chat_prompt.invoke({
    "assistant": "小助手",
    "now": "2024年1月",
    "hobby": "编程"
})

print("=== ChatPromptTemplate ===")
print("to_string():")
print(chat_value.to_string())
print()
print("to_messages():")
for msg in chat_value.to_messages():
    print(f"  - {type(msg).__name__}: {msg.content}")

输出：

=== ChatPromptTemplate ===
to_string():
System: 你是小助手，当前时间是 2024年1月
Human: 我喜欢编程，请推荐一部相关电影

to_messages():
  - SystemMessage: 你是小助手，当前时间是 2024年1月
  - HumanMessage: 我喜欢编程，请推荐一部相关电影

两者的核心区别：

方法返回内容返回类型适用场景
to_string()拼接好的纯文本，角色以 "System:" / "Human:" 前缀呈现str传给不支持消息角色的模型，如 text-davinci-003
to_messages()拆成多个消息对象，每个带角色和内容List[BaseMessage]传给支持聊天接口的模型，如 GPT-3.5-turbo、Claude

简单记法：

  • 字符串模板 + to_string() = 纯文本提示
  • 聊天模板 + to_messages() = 结构化消息列表

3. 聊天场景：用 ChatPromptTemplate​

跟聊天模型交互时，消息是有角色区分的。ChatPromptTemplate 就是为此设计的：

from langchain_core.prompts import (
    ChatPromptTemplate,
    HumanMessagePromptTemplate,
    MessagesPlaceholder,
)
from langchain_core.messages import AIMessage
from datetime import datetime

chat_prompt = ChatPromptTemplate.from_messages(
    [
        ("system", "你是OpenAI开发的聊天机器人,请根据用户的提问进行回复,当前时间为:{now}"),
        # 有时候可能还有其他消息,但是不确定
        MessagesPlaceholder("chat_history"),
        HumanMessagePromptTemplate.from_template("请讲一个关于{subject}的冷笑话"),
    ]
).partial(now=datetime.now)

chat_prompt_value = chat_prompt.invoke({
    "chat_history": [
        ("human", "我叫GuguLH"),
        AIMessage("你好,我是ChatGPT,有什么可以帮助您的?")
    ],
    "subject": "程序员",
})
print(chat_prompt_value)
print(chat_prompt_value.to_string())

几个关键点：

  • ChatPromptTemplate.from_messages() 接受元组列表，每个元组是 (角色, 内容) 格式
  • 常用角色：system（系统）、human（用户）、ai（助手）
  • MessagesPlaceholder("chat_history") 是占位符，运行时动态填入聊天历史，这招很实用
  • .partial(now=datetime.now) 可以预填充某些变量，不用每次调用都传。这里传的是函数本身 datetime.now，而不是 datetime.now() 的结果，这样每次调用时都会重新计算当前时间

4. 模板组合：用加号拼接​

多个模板可以拼接成一个，代码更模块化：

from langchain_core.prompts import ChatPromptTemplate

system_chat_prompt = ChatPromptTemplate.from_messages(
    [
        ("system", "你是OpenAI开发的聊天机器人,请根据用户的提问进行回复, 我叫{username}"),
    ]
)

human_chat_prompt = ChatPromptTemplate.from_messages(
    [
        ("human", "{query}")
    ]
)

chat_prompt = system_chat_prompt + human_chat_prompt
print(chat_prompt.invoke({
    "username": "GuguLH",
    "query": "你好,你是?",
}))

system_chat_prompt 定义了系统角色，human_chat_prompt 定义了用户输入，用加号拼在一起就是完整的聊天提示模板。这种写法比把所有东西写在一个大字符串里清晰多了。

5. 模板复用：拆成小块更灵活​

把提示词拆成多个独立模块，再组装成完整模板，好处是便于维护和复用：

from langchain_core.prompts import PromptTemplate

# 描述模板
instruction = "你正在模拟{person}"
# 示例模板
example = """
    下面是一个例子:
    Q:{example_q}
    A:{example_a}
    """

# 开始模板
start = """
    现在你是一个真实的人,请回答用户的问题:
    Q:{input}
    """

full_template = PromptTemplate.from_template(
    f"""
    {instruction}

    {example}

    {start}
    """
)

print(full_template.invoke({
    "person": "雷军",
    "example_q": "你最喜欢的汽车是什么?",
    "example_a": "小米su7",
    "input": "你最喜欢的手机是什么?",
}).to_string())

这个例子把提示词分成三部分：身份设定、示例问答、实际问答。组合时用 f-string 拼成完整模板。实际项目中可以把 instruction、example 这些做成可复用的函数。

6. 两个模板的关系​

记住两条线：

简单记忆：

模板输出
PromptTemplate纯文本字符串 / [HumanMessage]
ChatPromptTemplate带角色前缀文本 / [SystemMessage, HumanMessage, ...]

6.1 ChatPromptTemplate 的内部组成​

from_messages() 接收的列表里，每一项可以是不同类型：

举例：

chat_prompt = ChatPromptTemplate.from_messages([
    ("system", "你是{assistant}"),           # 字符串元组
    SystemMessage(content="固定内容"),       # 消息对象
    HumanMessagePromptTemplate.from_template("我叫{name}"),  # 模板对象
    MessagesPlaceholder("history"),         # 占位符，运行时填入
    "{query}"                               # 字符串，自动转 HumanMessagePromptTemplate
])

6.2 两种输出的对比​

记住这个对应关系：

调用方式场景
PromptTemplate + to_string()简单任务，模型只需要一段文字
PromptTemplate + to_messages()模拟单轮对话
ChatPromptTemplate + to_string()调试时打印查看
ChatPromptTemplate + to_messages()真实聊天模型调用

7. 几个常用的写法​

1. 字符串模板直接拼接

prompt = (
    PromptTemplate.from_template("你是{role}")
    + "，请回答以下问题："
    + "\n{question}"
)

2. 预填充部分变量

base_prompt = ChatPromptTemplate.from_messages([
    ("system", "当前日期是 {date}"),
    ("human", "{query}"),
]).partial(date="2024-01-01")

把固定不变的变量用 .partial() 预填，后续调用时就不用重复传了。

3. 动态聊天历史

chat_prompt = ChatPromptTemplate.from_messages([
    ("system", "你是助手"),
    MessagesPlaceholder("history"),
    ("human", "{input}"),
])

chat_prompt.invoke({
    "history": [("human", "你好"), ("ai", "你好，有什么可以帮你？")],
    "input": "今天天气怎么样",
})

用 MessagesPlaceholder 占住聊天历史的位置，调用时传入实际的消息列表。

Python 事件循环入门

2025年6月12日 · 阅读需 24 分钟
GuguLH
Back-end engineers

前言​

在上一节，我们讨论了 Python 异步编程。其中有一个核心概念就是事件循环（Event Loop）。 事件循环可以被想象成一个智能的“任务调度中心” ，它能够高效地管理和执行多个并发任务。它的工作原理是：当程序在等待（如等待 I/O 操作完成）某个任务时，事件循环会立即切换去处理其他已就绪的任务，而不是白白等待，从而大幅提升程序的执行效率。 本文将从 Python 初学者的角度出发，深入浅出地解析事件循环的工作原理、核心机制和实际应用，帮助你全面理解和掌握这个强大的异步工具。

1. 什么是事件循环？​

1.1 生动的比喻：餐厅服务员的智慧​

想象一下，你是一位餐厅的服务员。传统的同步方式就像你一次只能服务一桌客人：必须等第一桌客人点完餐、上完菜、结完账，才能去服务第二桌。如果某桌客人点餐很慢，其他客人就只能干等着。

而事件循环就像一位高效的服务员：当第一桌客人在看菜单时，你可以先去给第二桌上菜；当第二桌客人在用餐时，你可以去第三桌收钱；当第三桌客人需要加菜时，第一桌可能已经看完菜单了，你可以回去继续服务。你不断地在几桌客人之间切换，确保所有客人都能得到及时的服务，而不是傻傻地等待。

1.2 事件循环的本质​

在 Python 中，事件循环就是一个任务调度器，它的核心职责是：

  1. 管理任务队列：维护一个待执行的任务列表
  2. 调度任务执行：决定哪个任务在什么时候执行
  3. 处理等待事件：当任务需要等待（如网络请求、文件读取）时，切换到其他可执行的任务
  4. 响应事件完成：当等待的事件完成时，恢复对应的任务继续执行

事件循环的核心思想是：在等待时做其他事情，而不是让程序"傻等"。

1.3 事件循环的类型​

Python 的 asyncio 模块在不同操作系统上使用不同的事件循环实现：

  • Unix/Linux/macOS：使用 SelectorEventLoop，基于 selectors 模块（底层使用 epoll、kqueue 等）
  • Windows：使用 ProactorEventLoop，基于 I/O 完成端口（IOCP）

这些底层实现由 Python 自动选择，我们通常不需要关心。但了解这一点有助于理解事件循环的高效性。

2. 事件循环的工作机制​

2.1 事件循环的执行流程​

事件循环的工作可以概括为一个不断循环的过程：

2.2 任务队列和事件队列​

事件循环维护两个重要的队列：

  1. 任务队列（Task Queue）：存放准备执行的任务
  2. 就绪队列（Ready Queue）：存放已完成等待、可以继续执行的任务

事件循环会：

  • 优先执行就绪队列中的任务（这些任务之前被暂停，现在可以继续了）
  • 然后执行任务队列中的新任务
  • 当任务需要等待时，将其暂停，继续执行其他任务
  • 定期检查系统事件（如网络响应、文件读取完成），如果有事件完成，将对应的等待任务移入就绪队列

2.3 一个简单的例子​

让我们通过代码来理解事件循环的工作方式：

import asyncio
import time

async def task(name, duration):
    """模拟一个需要等待的任务"""
    print(f"[{time.time():.2f}] 任务 {name} 开始执行")
    await asyncio.sleep(duration)  # 等待 duration 秒
    print(f"[{time.time():.2f}] 任务 {name} 执行完成")

async def main():
    """主函数"""
    print("=== 事件循环开始工作 ===")

    # 创建三个任务，它们会被加入任务队列
    task1 = asyncio.create_task(task("A", 2))
    task2 = asyncio.create_task(task("B", 1))
    task3 = asyncio.create_task(task("C", 1.5))

    # 等待所有任务完成
    await asyncio.gather(task1, task2, task3)

    print("=== 所有任务完成 ===")

# 运行事件循环
asyncio.run(main())

运行这个程序，你会看到类似这样的输出：

=== 事件循环开始工作 ===
[123456.78] 任务 A 开始执行
[123456.78] 任务 B 开始执行
[123456.78] 任务 C 开始执行
[123457.78] 任务 B 执行完成
[123458.18] 任务 C 执行完成
[123458.78] 任务 A 执行完成
=== 所有任务完成 ===

关键观察：

  • 三个任务几乎同时开始（都在 123456.78 秒）
  • 任务 B 最先完成（等待 1 秒）
  • 任务 C 其次完成（等待 1.5 秒）
  • 任务 A 最后完成（等待 2 秒）
  • 总耗时约 2 秒，而不是 2 + 1 + 1.5 = 4.5 秒

这就是事件循环的威力：它让多个任务并发执行，而不是串行执行。

3. 事件循环的生命周期​

3.1 事件循环的创建、运行和关闭​

事件循环有明确的生命周期，理解这一点对于正确使用事件循环非常重要。

方式一：使用 asyncio.run()（推荐）​

asyncio.run() 是 Python 3.7+ 推荐的方式，它会自动处理事件循环的创建、运行和关闭：

import asyncio

async def main():
    print("主函数执行")
    await asyncio.sleep(1)

# asyncio.run() 做了以下事情：
# 1. 创建新的事件循环
# 2. 运行传入的协程
# 3. 等待所有任务完成
# 4. 关闭事件循环
asyncio.run(main())

方式二：手动管理事件循环​

在某些特殊场景下（如需要在已有事件循环中运行代码），可以手动管理：

import asyncio

async def main():
    print("主函数执行")
    await asyncio.sleep(1)

# 手动管理事件循环
loop = asyncio.get_event_loop()
try:
    # 运行直到完成
    loop.run_until_complete(main())
finally:
    # 关闭事件循环
    loop.close()

重要提示：

  • 每个线程只能有一个运行中的事件循环
  • 事件循环关闭后不能再次使用
  • 使用 asyncio.run() 是最简单、最安全的方式

3.2 获取当前事件循环​

在协程内部，可以使用 asyncio.get_running_loop() 获取当前运行的事件循环：

import asyncio

async def show_loop_info():
    """显示当前事件循环的信息"""
    loop = asyncio.get_running_loop()
    print(f"事件循环类型: {type(loop).__name__}")
    print(f"事件循环是否运行: {loop.is_running()}")
    print(f"事件循环是否关闭: {loop.is_closed()}")

async def main():
    await show_loop_info()

asyncio.run(main())

4. 事件循环的核心：调度算法​

4.1 如何决定执行哪个任务？​

事件循环使用一种叫做协作式多任务（Cooperative Multitasking）的调度方式：

  1. 任务主动让出控制权：任务通过 await 关键字主动暂停，让出执行权
  2. 事件循环选择下一个任务：从就绪队列或任务队列中选择一个可以执行的任务
  3. 任务恢复执行：当等待的事件完成时，任务被重新加入就绪队列

这种方式的优点是：

  • 高效：不需要线程切换的开销
  • 可控：任务明确知道何时让出控制权
  • 简单：不需要处理复杂的锁和同步问题

4.2 await 的作用机制​

await 关键字是事件循环调度的关键：

import asyncio

async def example():
    print("步骤 1: 开始执行")

    # await 的作用：
    # 1. 暂停当前任务
    # 2. 将控制权交还给事件循环
    # 3. 事件循环可以执行其他任务
    await asyncio.sleep(1)

    print("步骤 2: 恢复执行")

当执行到 await asyncio.sleep(1) 时：

  1. 当前任务被暂停
  2. 一个"等待 1 秒"的事件被注册到系统
  3. 事件循环继续执行其他任务
  4. 1 秒后，系统通知事件循环事件完成
  5. 任务被重新加入就绪队列
  6. 事件循环继续执行该任务，打印"步骤 2"

4.3 事件循环的单线程特性​

重要理解：事件循环在单线程中运行。这意味着：

  • 同一时刻只有一个任务在执行
  • 任务之间不会真正并行，而是并发（交替执行）
  • 不需要担心线程安全问题（在同一线程内）

但是，对于 I/O 操作（网络请求、文件读取），虽然任务本身在单线程中，但实际的 I/O 操作是由操作系统在后台完成的，所以可以实现真正的并发效果。

4.4 事件循环的底层机制​

事件循环的高效性来自于操作系统的 I/O 多路复用机制：

  • Linux：使用 epoll，可以高效地监控大量文件描述符
  • macOS/BSD：使用 kqueue，类似 epoll
  • Windows：使用 IOCP（I/O 完成端口）

这些机制允许事件循环在单线程中同时监控多个 I/O 操作，当任何一个操作完成时，操作系统会通知事件循环，事件循环就可以恢复对应的任务。

5. 实践：观察事件循环的工作过程​

5.1 创建一个简单的事件循环观察器​

让我们写一个程序来观察事件循环如何调度任务：

import asyncio
import time

async def worker(name, work_time):
    """模拟一个工作协程"""
    print(f"[{time.time():.2f}] 工人 {name}: 开始工作")

    # 模拟工作过程，分几个步骤
    for i in range(3):
        await asyncio.sleep(work_time / 3)
        print(f"[{time.time():.2f}] 工人 {name}: 完成步骤 {i+1}/3")

    print(f"[{time.time():.2f}] 工人 {name}: 工作完成")

async def main():
    """主函数：观察事件循环如何调度多个工人"""
    print("=== 事件循环：任务调度演示 ===\n")

    start_time = time.time()

    # 创建三个工人任务
    # 工人A需要3秒，工人B需要2秒，工人C需要1秒
    await asyncio.gather(
        worker("A", 3),
        worker("B", 2),
        worker("C", 1)
    )

    end_time = time.time()
    print(f"\n总耗时: {end_time - start_time:.2f} 秒")
    print("（如果是串行执行，需要 3+2+1=6 秒）")

asyncio.run(main())

运行这个程序，你会清楚地看到：

  • 三个工人同时开始工作
  • 他们交替完成各个步骤
  • 最终总耗时约等于最慢工人的时间（3秒）

5.2 事件循环的优先级​

事件循环按照以下优先级处理任务：

  1. 就绪队列中的任务：之前被暂停，现在可以继续执行
  2. 新创建的任务：刚被创建，还未开始执行
  3. 等待中的任务：正在等待事件完成，暂时不执行
import asyncio

async def task(name, delay):
    print(f"任务 {name} 开始")
    await asyncio.sleep(delay)
    print(f"任务 {name} 完成")

async def main():
    # 创建任务
    t1 = asyncio.create_task(task("A", 2))
    t2 = asyncio.create_task(task("B", 1))
    t3 = asyncio.create_task(task("C", 0.5))

    # 等待所有任务
    await asyncio.gather(t1, t2, t3)

asyncio.run(main())

执行顺序可能是：C 最先完成，然后是 B，最后是 A。事件循环会优先处理能够继续执行的任务。

6. 异常处理和错误管理​

6.1 事件循环中的异常处理​

在事件循环中，异常处理非常重要。如果一个任务抛出异常而没有处理，可能会影响整个程序。

import asyncio

async def risky_task(name, should_fail=False):
    """可能失败的任务"""
    print(f"任务 {name} 开始")
    await asyncio.sleep(0.5)

    if should_fail:
        raise ValueError(f"任务 {name} 执行失败")

    print(f"任务 {name} 完成")
    return f"任务 {name} 的结果"

async def safe_task(name, should_fail=False):
    """带异常处理的安全任务"""
    try:
        print(f"任务 {name} 开始")
        await asyncio.sleep(0.5)

        if should_fail:
            raise ValueError(f"任务 {name} 执行失败")

        print(f"任务 {name} 完成")
        return f"任务 {name} 的结果"
    except Exception as e:
        print(f"任务 {name} 发生错误: {e}")
        return None

async def main():
    print("=== 异常处理演示 ===\n")

    # 方式1: 使用 try-except 包装 gather
    try:
        results = await asyncio.gather(
            risky_task("A", False),
            risky_task("B", True),  # 这个会失败
            risky_task("C", False),
            return_exceptions=True  # 重要：不中断其他任务
        )
        print(f"\n结果: {results}")
    except Exception as e:
        print(f"捕获到异常: {e}")

    print("\n" + "="*50 + "\n")

    # 方式2: 在任务内部处理异常（推荐）
    results = await asyncio.gather(
        safe_task("D", False),
        safe_task("E", True),  # 这个会失败但被处理
        safe_task("F", False)
    )
    print(f"\n结果: {results}")

asyncio.run(main())

关键点：

  • 使用 return_exceptions=True 可以让 gather() 不因单个任务失败而中断
  • 在任务内部处理异常是更好的做法
  • 未处理的异常会导致任务失败，但不会影响其他任务

6.2 超时处理​

事件循环还支持超时机制，避免任务无限等待：

import asyncio

async def slow_task(name, duration):
    """模拟一个慢任务"""
    print(f"任务 {name} 开始，预计需要 {duration} 秒")
    await asyncio.sleep(duration)
    print(f"任务 {name} 完成")
    return f"任务 {name} 的结果"

async def main():
    try:
        # 设置 2 秒超时
        result = await asyncio.wait_for(slow_task("A", 3), timeout=2.0)
        print(f"结果: {result}")
    except asyncio.TimeoutError:
        print("任务超时！")

asyncio.run(main())

7. 常见陷阱和注意事项​

7.1 陷阱一：在事件循环中调用阻塞函数​

错误示例：

import asyncio
import time

async def bad_example():
    print("开始")
    time.sleep(2)  # 错误！这是阻塞调用
    print("结束")

async def main():
    await asyncio.gather(
        bad_example(),
        bad_example()
    )

asyncio.run(main())

这段代码会串行执行，因为 time.sleep() 是阻塞的，会阻止事件循环切换任务。

正确做法：

import asyncio

async def good_example():
    print("开始")
    await asyncio.sleep(2)  # 正确！使用异步版本
    print("结束")

async def main():
    await asyncio.gather(
        good_example(),
        good_example()
    )

asyncio.run(main())

7.2 陷阱二：忘记使用 await​

错误示例：

import asyncio

async def task():
    await asyncio.sleep(1)
    return "完成"

async def main():
    result = task()  # 错误！没有使用 await
    print(result)  # 输出: <coroutine object task at 0x...>

asyncio.run(main())

正确做法：

import asyncio

async def task():
    await asyncio.sleep(1)
    return "完成"

async def main():
    result = await task()  # 正确！
    print(result)  # 输出: 完成

asyncio.run(main())

7.3 陷阱三：事件循环的嵌套问题​

在已经运行的事件循环中，不能再次调用 asyncio.run()：

import asyncio

async def inner_task():
    # 错误：不能在运行中的事件循环里再次运行
    # asyncio.run(some_other_task())  # 这会报错

    # 正确：直接 await 或使用 create_task
    await asyncio.sleep(1)

async def main():
    await inner_task()

asyncio.run(main())

7.4 陷阱四：CPU 密集型任务阻塞事件循环​

事件循环不适合 CPU 密集型任务，因为这类任务不会主动让出控制权：

import asyncio

def cpu_intensive_task():
    """CPU 密集型任务"""
    result = 0
    for i in range(10000000):
        result += i
    return result

async def bad_async_cpu_task():
    # 这会阻塞事件循环
    result = cpu_intensive_task()
    return result

# 正确做法：使用线程池或进程池
async def good_async_cpu_task():
    loop = asyncio.get_event_loop()
    # 在线程池中执行 CPU 密集型任务
    result = await loop.run_in_executor(None, cpu_intensive_task)
    return result

8. 实际应用示例​

8.1 并发处理多个网络请求​

让我们写一个实用的例子：同时获取多个网页的内容。

import asyncio
import aiohttp
import time

async def fetch_url(session, url):
    """异步获取网页内容"""
    try:
        print(f"[{time.time():.2f}] 开始获取 {url}")
        async with session.get(url, timeout=aiohttp.ClientTimeout(total=5)) as response:
            content = await response.text()
            print(f"[{time.time():.2f}] 完成获取 {url}, 长度: {len(content)} 字符")
            return {"url": url, "length": len(content), "status": "success"}
    except asyncio.TimeoutError:
        print(f"[{time.time():.2f}] 获取 {url} 超时")
        return {"url": url, "status": "timeout"}
    except Exception as e:
        print(f"[{time.time():.2f}] 获取 {url} 失败: {e}")
        return {"url": url, "status": "error", "error": str(e)}

async def main():
    """主函数"""
    urls = [
        "https://www.python.org",
        "https://www.github.com",
        "https://www.stackoverflow.com"
    ]

    start = time.time()

    # 创建 HTTP 会话
    async with aiohttp.ClientSession() as session:
        # 同时获取所有网页
        results = await asyncio.gather(
            *[fetch_url(session, url) for url in urls],
            return_exceptions=True
        )

    end = time.time()

    print(f"\n=== 结果汇总 ===")
    print(f"总共获取 {len(results)} 个网页")
    print(f"总耗时: {end - start:.2f} 秒")
    for result in results:
        if isinstance(result, dict):
            print(f"  {result['url']}: {result['status']}")

# 运行程序
# 注意：需要安装 aiohttp: pip install aiohttp
if __name__ == "__main__":
    asyncio.run(main())

8.2 控制并发数量​

有时候，我们不想同时运行太多任务（比如避免服务器压力过大）。可以使用信号量（Semaphore）来控制并发数量：

import asyncio
import time

async def worker(semaphore, task_id, work_time):
    """工作协程"""
    async with semaphore:  # 获取信号量
        print(f"[{time.time():.2f}] 任务 {task_id} 开始")
        await asyncio.sleep(work_time)  # 模拟工作
        print(f"[{time.time():.2f}] 任务 {task_id} 完成")

async def main():
    """主函数"""
    # 最多同时运行 3 个任务
    semaphore = asyncio.Semaphore(3)

    # 创建 10 个任务
    tasks = [
        worker(semaphore, i, 2)
        for i in range(10)
    ]

    start = time.time()
    await asyncio.gather(*tasks)
    end = time.time()

    print(f"\n总耗时: {end - start:.2f} 秒")
    print("（如果串行执行，需要 10 * 2 = 20 秒）")

asyncio.run(main())

运行这个程序，你会发现最多只有 3 个任务同时执行，但总耗时远少于串行执行。

9. 事件循环的应用场景​

9.1 什么时候使用事件循环？​

事件循环特别适合以下场景：

  1. I/O 密集型任务：网络请求、文件读写、数据库查询
  2. 需要并发处理多个请求：Web 服务器处理多个客户端请求
  3. 定时任务：需要定期执行某些操作
  4. 实时应用：聊天应用、实时数据推送
  5. 爬虫程序：需要同时抓取多个网页

9.2 不适合的场景​

事件循环不适合：

  1. CPU 密集型任务：大量计算、图像处理（这些任务不会主动让出控制权，会阻塞事件循环）
  2. 需要真正并行执行的任务：需要使用多进程或多线程
  3. 简单的单任务程序：如果只有一个任务，使用同步代码更简单

对于 CPU 密集型任务，应该使用：

  • 多进程：multiprocessing 模块
  • 线程池：concurrent.futures.ThreadPoolExecutor
  • 在事件循环中使用执行器：loop.run_in_executor()

10. 总结与核心要点​

事件循环是 Python 异步编程的核心机制，理解它对于编写高效的并发程序至关重要。

核心概念描述
任务队列事件循环维护任务队列，存放待执行的任务。
就绪队列存放已完成等待、可以继续执行的任务。
协作式调度任务通过 await 主动让出控制权，事件循环选择下一个任务执行。
单线程并发事件循环在单线程中运行，通过任务切换实现并发效果。
事件驱动当等待的事件完成时，对应的任务被恢复执行。
高效 I/O在等待 I/O 操作时，可以执行其他任务，充分利用时间。
生命周期管理使用 asyncio.run() 自动管理事件循环的创建、运行和关闭。
异常处理在任务内部处理异常，或使用 return_exceptions=True。

关键要点​

  1. 事件循环是单线程的：任务交替执行，不是真正并行
  2. await 是关键：它让任务暂停，让出控制权给事件循环
  3. 适合 I/O 密集型：不适合 CPU 密集型任务
  4. 避免阻塞调用：在事件循环中不要使用阻塞函数
  5. 正确处理异常：确保异常不会影响其他任务

前言​

在 Python 开发中，我们经常遇到需要等待的操作：读取文件、请求 API、查询数据库。传统同步代码会"卡住"，直到操作完成。异步编程（Async Programming）让程序在等待时继续处理其他任务，提升效率。Python 的 async/await 让异步编程更直观。本文面向初学者，用简单例子介绍异步编程的核心概念与实践。

1. 同步与异步：一个生动的比喻​

1.1 洗衣服的故事​

假设你要洗 3 批衣服，每批需要 30 分钟：

同步方式（传统编程）：

第1批：放入洗衣机 → 等待30分钟 → 取出 →
第2批：放入洗衣机 → 等待30分钟 → 取出 →
第3批：放入洗衣机 → 等待30分钟 → 取出
总时间：90分钟

在等待时，你只能干等，无法做其他事。

异步方式（异步编程）：

第1批：放入洗衣机1 → 启动
第2批：放入洗衣机2 → 启动
第3批：放入洗衣机3 → 启动
→ 等待期间可以做其他事（看书、做饭）→
30分钟后，所有衣服都洗好了
总时间：30分钟

在等待时，你可以做其他事，效率更高。

1.2 编程中的对应关系​

在编程中：

  • 同步：代码按顺序执行，遇到耗时操作会阻塞，直到完成
  • 异步：遇到耗时操作时，程序可以暂停该任务，先处理其他任务，等耗时操作完成后再继续

异步特别适合 I/O 密集型任务（网络请求、文件读写、数据库查询），因为大部分时间在等待外部响应，而不是计算。

2. async/await 入门：两个关键词​

2.1 async def：声明异步函数​

在 Python 中，使用 async def 定义异步函数。所有异步编程都需要导入 asyncio 模块：

# 所有异步编程都需要导入 asyncio 模块
import asyncio

async def download_file(filename):
    """模拟下载文件需要等待的操作"""
    print(f"开始下载 {filename}...")
    await asyncio.sleep(2)  # 模拟等待2秒
    print(f"{filename} 下载完成！")
    return f"{filename}的内容"

重要特点：

  • async def 定义的函数是协程函数（Coroutine Function）
  • 调用它不会立即执行，而是返回一个协程对象
  • 需要用 await 调用，或在事件循环中运行

2.2 await：等待异步操作完成​

await 用于等待异步操作完成：

async def main():
    # 使用 await 等待异步函数完成
    result = await download_file("文档.pdf")
    print(f"获取到结果：{result}")

关键理解：

  • await 只能在 async def 函数内使用
  • await 会暂停当前函数，让其他任务运行
  • 等待期间，程序可以处理其他任务

2.3 asyncio.run()：启动异步程序​

异步函数需要在事件循环（Event Loop）中运行。事件循环就像一个任务调度器，负责管理和执行所有的异步任务，让程序在等待某个任务时能够切换到其他任务。

最简单的方式是使用 asyncio.run()：

# 运行异步主函数
asyncio.run(main())

这行代码会创建事件循环，运行异步函数，并在完成后关闭事件循环。

3. 实践案例：模拟多用户登录验证​

下面是一个实用的例子：模拟多个用户同时进行登录验证。

3.1 同步版本（效率较低）​

import time

def verify_user_login(username):
    """模拟验证用户登录需要1秒"""
    print(f"正在验证 {username}...")
    time.sleep(1)  # 模拟网络请求延迟
    print(f"{username} 验证完成")
    return f"{username}登录成功"

def process_users_sync():
    user_list = ["张三", "李四", "王五", "赵六"]

    start_time = time.time()
    results = []

    # 逐个处理，必须等上一个完成才能处理下一个
    for username in user_list:
        result = verify_user_login(username)
        results.append(result)

    total_time = time.time() - start_time
    print(f"\n同步方式总耗时：{total_time:.2f} 秒")
    return results

# 运行同步版本
process_users_sync()

输出示例：

正在验证 张三...
张三 验证完成
正在验证 李四...
李四 验证完成
正在验证 王五...
王五 验证完成
正在验证 赵六...
赵六 验证完成

同步方式总耗时：4.01 秒

这种方式会逐个处理，总耗时等于每个操作的时间之和。

3.2 异步版本（效率更高）​

import asyncio
import time

async def verify_user_login_async(username):
    """异步版本的登录验证"""
    print(f"正在验证 {username}...")
    await asyncio.sleep(1)  # 使用异步等待，不会阻塞
    print(f"{username} 验证完成")
    return f"{username}登录成功"

async def process_users_async():
    user_list = ["张三", "李四", "王五", "赵六"]

    start_time = time.time()

    # 创建多个任务，并发执行
    tasks = [verify_user_login_async(username) for username in user_list]

    # 等待所有任务完成
    results = await asyncio.gather(*tasks)

    total_time = time.time() - start_time
    print(f"\n异步方式总耗时：{total_time:.2f} 秒")
    return results

# 运行异步版本
asyncio.run(process_users_async())

输出示例：

正在验证 张三...
正在验证 李四...
正在验证 王五...
正在验证 赵六...
张三 验证完成
李四 验证完成
王五 验证完成
赵六 验证完成

异步方式总耗时：1.01 秒

关键改进：

  • 所有验证任务同时启动
  • 总耗时接近单个操作的时间（约 1 秒）
  • 效率明显提升

3.3 代码解析：asyncio.gather() 的作用​

asyncio.gather(*tasks) 的作用：

  • 并发执行：同时启动多个异步任务
  • 等待完成：等待所有任务完成
  • 返回结果：返回所有任务的结果列表，结果顺序与输入顺序一致（即使任务完成顺序不同）

*tasks 是 Python 的解包语法，将列表展开为多个参数传递给 gather()。

4. 另一个例子：模拟批量处理文件​

让我们看另一个例子：模拟批量读取和处理文件。

import asyncio
import time

async def process_single_file(filename, process_time):
    """模拟处理文件需要的时间"""
    print(f"开始处理 {filename}...")
    await asyncio.sleep(process_time)  # 模拟处理时间
    print(f"{filename} 处理完成")
    return f"{filename}已处理"

async def batch_process_files():
    file_list = [
        ("文件1.txt", 0.5),
        ("文件2.txt", 0.8),
        ("文件3.txt", 0.3),
        ("文件4.txt", 0.6)
    ]

    start_time = time.time()

    # 使用 asyncio.gather() 并发处理
    tasks = [
        process_single_file(filename, process_time)
        for filename, process_time in file_list
    ]
    results = await asyncio.gather(*tasks)

    total_time = time.time() - start_time
    print(f"\n总耗时：{total_time:.2f} 秒")
    print(f"结果：{results}")

    return results

# 运行
asyncio.run(batch_process_files())

输出示例：

开始处理 文件1.txt...
开始处理 文件2.txt...
开始处理 文件3.txt...
开始处理 文件4.txt...
文件3.txt 处理完成
文件1.txt 处理完成
文件4.txt 处理完成
文件2.txt 处理完成

总耗时：0.80 秒
结果：['文件1.txt已处理', '文件2.txt已处理', '文件3.txt已处理', '文件4.txt已处理']

注意：总耗时是 0.80 秒（最慢的文件处理时间），而不是所有时间的总和（2.2 秒）。

5. 常见应用场景​

异步编程特别适合以下场景：

  1. 网络爬虫：同时抓取多个网页
  2. API 调用：并发调用多个外部 API
  3. 数据库查询：同时执行多个查询（使用异步数据库库如 aiomysql）
  4. 文件处理：并发读取多个文件（使用 aiofiles）
  5. Web 服务器：处理大量并发请求（如 FastAPI）
  6. 实时通信：聊天应用、WebSocket 服务器

注意：使用异步库需要额外安装：

  • 网络请求：pip install aiohttp
  • 文件操作：pip install aiofiles
  • 数据库：pip install aiomysql（MySQL）或 pip install asyncpg（PostgreSQL）

重要提醒：异步编程主要提升 I/O 密集型任务的效率。对于 CPU 密集型任务（如复杂计算、图像处理），异步帮助有限，可能需要多进程。

6. 常见陷阱与注意事项​

6.1 陷阱一：忘记使用 await​

# 错误示例
async def wrong_example():
    result = process_single_file("文件.txt", 1)  # 忘记 await
    print(result)  # 打印的是协程对象，不是结果！

# 正确示例
async def correct_example():
    result = await process_single_file("文件.txt", 1)  # 使用 await
    print(result)  # 打印实际结果

6.2 陷阱二：在普通函数中使用 await​

# 错误：普通函数不能使用 await
def normal_function():
    await process_single_file("文件.txt", 1)  # 语法错误！

# 正确：必须在异步函数中使用
async def async_function():
    await process_single_file("文件.txt", 1)

6.3 陷阱三：混用同步阻塞操作​

# 错误：在异步函数中使用同步阻塞操作
import time
async def wrong_way():
    time.sleep(1)  # 这会阻塞整个事件循环！

# 正确：使用异步等待
import asyncio
async def correct_way():
    await asyncio.sleep(1)  # 不会阻塞，其他任务可以运行

7. 总结与最佳实践​

Python 的 async/await 让异步编程更直观。掌握以下要点：

实践要点说明
理解核心概念async def 定义异步函数，await 等待异步操作完成
使用 asyncio.run()运行异步主函数的标准方式
并发执行任务使用 asyncio.gather() 实现多个任务的并发执行
选择合适的场景异步适合 I/O 密集型任务，不适合 CPU 密集型任务
使用异步库网络请求用 aiohttp，文件操作用 aiofiles，避免混用同步阻塞操作
错误处理在异步函数中使用 try/except 处理异常，与同步代码类似

核心思想：异步编程让程序在等待时继续处理其他任务，就像同时使用多台洗衣机洗衣服，而不是等一台洗完再用下一台。掌握 async/await 后，你就能编写更高效的 Python 程序，特别是在处理网络请求和 I/O 操作时。
